# DEVLOG

## 文档目的

这份文档面向后续继续维护 `RoyalFight` 原型的开发者。
它不重复用户说明文档里的基础玩法介绍，而是重点记录：
- 这个原型是怎么一步步搭起来的。
- 当前代码结构为什么这样组织。
- 已经遇到过哪些稳定性问题。
- 后面继续迭代时，最值得优先处理什么。

更新日期：2026-03-26

## 当前项目定位

`RoyalFight` 是一个基于 Godot 4.6 的《皇室战争》风格双路实时对战原型。
目标是先把"可玩、可调试、可扩展"的核心战斗循环跑通，再逐步补表现和内容。

当前更像是一个"战斗实验场"，而不是完整产品版本。

## 迭代过程简述

### 1. 项目初始化

最初版本完成了最小可运行结构：
- `project.godot`
- `Main.tscn`
- `scripts/main.gd`
- `scripts/battle_unit.gd`
- `scripts/battle_building.gd`

主场景使用脚本驱动生成战场和 UI，没有先拆成大量场景资源。
这样做的目的是先快速建立玩法闭环，减少前期资源组织成本。

### 2. 核心战斗循环

第一阶段完成的内容：
- 四张基础卡牌。
- 圣水恢复。
- 玩家部署。
- 敌方自动出兵。
- 单位自动推进和攻击。
- 左右公主塔、国王塔与胜负判定。

这时的版本已经可以完整对局，但路径、战斗和观感都比较粗糙。

### 3. 双路桥梁推进

后续把战场逻辑改成了更接近《皇室战争》的双路结构：
- 左右路明确分离。
- 单位按兵线经过桥梁推进。
- 左右公主塔优先影响各自兵线。
- 战场增加了中线、河道和桥梁的可视反馈。

这一阶段的主要改动集中在：
- `scripts/main.gd`
- `scripts/battle_unit.gd`
- `scripts/battle_building.gd`

### 4. 塔唤醒、近战卡位、阻挡

为了让战斗读起来更像《皇室战争》，加入了：
- 国王塔默认休眠。
- 公主塔被摧毁或国王塔受击后唤醒。
- 近战单位贴脸停下输出。
- 同路友军出现前后堵塞。
- 单位之间做轻微分离。

这一步让"推进感"和"桥头会战感"明显更接近目标体验。

### 5. 仇恨切换与近战绕行

继续往真实战斗手感上靠，补了：
- 受击后的目标记忆。
- 附近更优目标的重新切换。
- 前排堵住时的简单侧向绕行。

当前这一套逻辑仍然是轻量方案，不是完整导航或群体寻路系统。

### 6. 攻击前摇 / 后摇 / 受击停顿

为了让交战不再像"贴上去就瞬间结算"，补了：
- 攻击前摇。
- 后摇恢复。
- 受击硬直。
- 轻微击退。
- 绘制上的压缩、拉伸和高亮反馈。

这部分主要在单位和塔的 `_draw()` 中做视觉表现，在 `_process()` 中做状态计时。

### 7. 音效与美术替换

在表现层上又做了两件事：
- 增加程序化攻击音效和命中音效。
- 用 SVG 素材替换原本的圆形 / 方块占位图。

SVG 不是直接作为普通资源 `load()` 使用，而是运行时读取文本后转成 `ImageTexture`。
原因是当前环境下直接 `load("res://xxx.svg")` 会出现资源加载失败。

### 8. 手牌系统

实现了真正的 4 手牌轮转机制：
- `player_hand`: 当前4张手牌（有序数组）
- `player_deck`: 完整卡组（8张 = 2倍卡牌池）
- `next_card_preview`: 下一张预览

**轮转机制**：
- 游戏开始时从卡组抽取4张作为手牌
- 出牌后从牌库抽1张补充到手牌末尾
- 牌库为空时自动重新洗牌

**UI更新**：
- 底部4个手牌槽从左到右显示当前手牌
- 右侧"下一张"预览框显示即将抽到的卡牌
- 出牌后手牌自动轮转

### 9. 特效优化与队伍区分

- 添加 `_visual_dirty` 标志优化重绘逻辑，解决特效闪烁问题
- 添加敌我双方队伍颜色边框环，区分己方（蓝色）和敌方（红色）单位

### 10. 弹道可视化

为远程单位（Archer）和箭塔添加了抛物线弹道效果：
- 弧形轨迹 + 带箭头的箭矢形状
- 己方亮黄色，敌方橙色

## 当前代码结构

### `scripts/main.gd`

职责：
- 战场常量与卡牌定义。
- UI 创建与更新。
- 玩家部署和敌方 AI 出兵。
- 路径点计算。
- 单位和建筑索敌辅助。
- 命中音效 / 攻击音效。
- SVG 运行时栅格化与缓存。
- 实体销毁、国王塔唤醒、整局胜负流程。
- 手牌系统管理。

它现在是项目里的"主控制器"。
随着项目继续增长，未来最值得拆分的就是它。

### `scripts/battle_unit.gd`

职责：
- 单位基础属性。
- 索敌与攻击流程。
- 路径推进。
- 近战堵塞与绕行。
- 受击、击退、硬直。
- 弹道飞行逻辑。
- 单位自身绘制。

现在它本质上是一个"轻量战斗 Actor"。

### `scripts/battle_building.gd`

职责：
- 塔的受击与攻击。
- 国王塔休眠 / 唤醒后的行为。
- 弹道飞行逻辑。
- 建筑自身绘制。
- 对死亡目标的攻击引用清理。

### `assets/units/*.svg` 与 `assets/buildings/*.svg`

这些资源用于：
- 战场单位显示。
- 战场塔显示。
- 底部手牌按钮显示。

所有 SVG 都经过 `main.gd` 中的缓存加载函数统一处理。

## 关键实现决策

### 1. 先用脚本绘制，不先做完整节点层级

当前单位和塔大量依赖 `_draw()` 做表现，而不是一开始就拆成复杂场景。
优点：改玩法快、调参数快、原型阶段成本低。
缺点：后续接正式美术和动画时，需要重构一部分表现层。

### 2. SVG 运行时转纹理

当前使用 `load_svg_texture()`：
- 读取 SVG 文本。
- 用 `Image.load_svg_from_string()` 栅格化。
- 生成 `ImageTexture`。
- 放进 `_svg_texture_cache` 复用。

这样避免了直接 `load()` SVG 的兼容问题，也减少了重复生成纹理的成本。

### 3. 卡牌样式缓存

UI 更新最初是每一帧都创建新的 `StyleBoxFlat` 并覆盖到按钮上。
这会不断制造对象分配，运行时间长了以后容易导致明显卡顿。

现在改成：
- 用 `_card_style_cache` 缓存样式对象。
- 只在按钮状态变化时更新样式。

这是当前版本里一个很关键的性能修复点。

### 4. 实体销毁后的目标清理

这是最近一轮稳定性修复的重点。
之前在"塔被拆掉"的时刻，常出现：
- 单位还持有旧的 `current_target`。
- `_pending_target` 已经指向被释放对象。
- 命中结算后继续访问已经失效的目标。

当前修复方式包括：
- `battle_unit.gd` 和 `battle_building.gd` 在 `_process()` 里先清理失效目标引用。
- `main.gd` 的 `on_entity_destroyed()` 会广播给全场实体，让它们主动清掉指向死亡对象的引用。
- `_resolve_attack()` 结算时先抓 `resolved_target` 快照，避免回调后引用被清空。
- `on_damage_dealt()` 增加空值与有效性保护。

这部分是目前项目稳定性最需要保留的经验。

### 5. 视觉重绘优化

通过 `_visual_dirty` 标志控制 `queue_redraw()` 调用，避免每帧无条件重绘造成的闪烁问题。

## 最近修过的重要问题

### 1. 攻击前摇卡住不出手

早期问题：
- 前摇结束后没有稳定进入真正伤害结算。
- 单位和塔会贴住目标却不攻击。

解决方向：
- 调整攻击结算顺序。
- 最后改成更稳定的状态推进方式。

### 2. 塔被拆时的空引用中断

现象：
- 一座塔被拆掉后，调试器中断，看起来像卡死。

根因：
- 旧目标引用继续被访问。
- 命中回调和销毁回调之间存在时序竞争。

当前已做的保护：
- 失效引用清理。
- 销毁广播清理。
- 命中结算目标快照。
- 空目标保护。

### 3. UI 随时间越来越卡

根因：
- `_update_ui()` 每帧都在创建按钮样式对象。

现状：
- 已修成缓存样式。

### 4. 手牌出牌后不补充

根因：
- `_draw_one_card()` 中当牌库为空时调用了 `_setup_deck()`，该函数会重置整个牌组并重新抽4张手牌，覆盖了当前手牌。

现状：
- 已修复为只重新构建和洗牌库，不重置手牌。

### 5. Android Typed Array 兼容性

根因：
- Godot 4.x 在 Android 上对 typed array（`Array[String]`）的类型验证更严格，`pop_front()` 返回 `Nil` 会导致报错。

现状：
- 将 `player_deck` 和 `player_hand` 改为 untyped array。
- 使用 `shuffle()` 内置方法替代手动 shuffle。

## 当前已知限制

### 1. 全场实体扫描仍然偏重

目前很多逻辑仍然依赖：
- `get_tree().get_nodes_in_group("battle_entity")`

这出现在：
- 索敌。
- 建筑选目标。
- 单位分离。
- 友军阻挡。
- 胜负检查。

单位数量一旦显著增加，这会成为主要性能瓶颈。

### 2. AI 仍然是基础随机部署

当前 AI 只会：
- 检查可支付卡牌。
- 随机选卡。
- 随机选左右路。

它没有做：
- 反制逻辑。
- 节奏控制。
- 攻守判断。
- 卡差与费用管理。

### 3. 表现层仍然是原型级

现在虽然已经比最初的圆形占位图好很多，但还没有：
- 正式角色动画。
- 攻击动作资源。
- 死亡特效。
- 更强的河道 / 地形 / 阴影细节。

### 4. Android SVG 资源加载

Android 导出的 APK 仍存在 SVG 资源加载问题。
建议后续将 SVG 转换为 PNG 格式导出。

## 建议的下一步开发顺序

### 优先级 1：性能

先做：
- 实体列表缓存，而不是频繁 `get_nodes_in_group()`。
- 更明确的单位管理器。
- 分离"单位列表""建筑列表""可攻击目标列表"。

### 优先级 2：内容扩展

再往后可以做：
- 溅射单位。
- 群体召唤单位。
- 冲锋或跳跃型单位。
- 更复杂的塔和法术机制。

### 优先级 3：AI 增强

补上：
- 反制逻辑。
- 节奏控制。
- 攻守判断。
- 卡差与费用管理。

## 开发建议

后续改动时，建议遵守这些原则：
- 先保战斗闭环，再补表现。
- 每次改"攻击 / 销毁 / 目标引用"相关逻辑后，都做 30 秒以上长跑验证。
- 每次改 UI 更新逻辑时，注意不要重新引入"每帧创建对象"的问题。
- 如果开始补大量新单位，优先做实体管理和性能整理，不然后面会越来越难调。

## 当前验证状态

截至 2026-03-26，这个版本已经做过多轮 Godot MCP 长时间启动验证。
最近一轮针对"塔被拆后卡死"的修复，已经在 65 秒级别运行中通过，没有新的调试器中断。

后续如果再出现"运行久了卡住"，优先检查：
- 是否又出现了死亡实体引用残留。
- 是否有新的每帧对象分配。
- 是否是全场扫描导致的性能问题。

## 待解决问题

### Android SVG 资源加载

截至本次更新，Android 导出的 APK 仍存在 SVG 资源加载问题。
建议后续方案：
1. 在 Godot 编辑器中将 SVG 转换为 PNG
2. 更新代码使用 PNG 路径
3. 或者确保导出设置正确，重新导入所有资源后导出
