# DEVLOG

更新日期：2026-03-26

## 文档目的

这份文档面向后续继续维护 `RoyalFight` 原型的开发者。它不重复用户说明文档里的基础玩法介绍，而是重点记录：
- 这个原型是怎么一步步搭起来的。
- 当前代码结构为什么这样组织。
- 已经遇到过哪些稳定性和平台兼容问题。
- 接下来最值得优先处理的方向是什么。

## 当前项目定位

`RoyalFight` 是一个基于 Godot 4.6 的《皇室战争》风格双路实时对战原型。项目目标不是先做内容量，而是先把可玩、可调试、可扩展的核心战斗循环跑通，再逐步补表现、平台兼容和内容深度。

当前版本更像一个战斗实验场，而不是完整产品。

## 迭代过程简记

### 1. 原型起步

最早期只保留最小可运行结构：
- `project.godot`
- `Main.tscn`
- `scripts/main.gd`
- `scripts/battle_unit.gd`
- `scripts/battle_building.gd`

主场景主要依赖脚本动态生成战场和 UI，目的是快速建立玩法闭环，减少前期资源组织成本。

### 2. 核心战斗循环

第一阶段完成了这些内容：
- 玩家部署。
- 敌方自动出兵。
- 圣水回复。
- 单位自动推进和攻击。
- 公主塔、国王塔和胜负判定。

这一版已经可玩，但路径、卡位和观感都比较粗糙。

### 3. 双路、桥梁和兵线

后面把战场改造成更像《皇室战争》的双路结构：
- 左右路分离。
- 河道和桥梁。
- 单位按兵线推进。
- 左右塔更偏向本路交战。

这一阶段的核心改动集中在：
- `scripts/main.gd`
- `scripts/battle_unit.gd`
- `scripts/battle_building.gd`

### 4. 卡位、阻挡、仇恨切换

为了让战斗更像真实会战，又逐步加入：
- 国王塔休眠 / 唤醒。
- 近战贴脸后停下输出。
- 同路友军前后堵塞。
- 单位间简单分离。
- 受击后的目标记忆和转火。
- 堵路时的轻量绕行。

这套方案依旧是轻量逻辑，不是完整导航系统，但对原型手感提升很大。

### 5. 攻击动画曲线与受击表现

`battle_unit.gd` 和 `battle_building.gd` 中加入了：
- 攻击前摇。
- 后摇恢复。
- 受击停顿。
- 轻微击退。
- `_draw()` 内的压缩、拉伸、回摆和高亮反馈。

后续又修过一次数值过冲问题，避免受击 / 收招时单位异常放大或闪烁。

### 6. 远程弹道与命中解算

为了让 Archer 和塔的攻击表现更自然，加入了抛物线弹道可视化。后面又修正了解算逻辑：
- 投射物表现和真实命中目标引用分离。
- 不再依赖会被提前清掉的 `_pending_target`。
- 解决了“箭飞出去但不掉血”的问题。

### 7. 手牌系统与移动端适配

后续加上了更完整的 4 手牌系统：
- `player_hand` 表示当前 4 手牌。
- `player_deck` 表示抽牌池。
- `next_card_preview` 表示下一张预览。

最近一轮又补了：
- 手机触屏部署支持。
- 窄屏下手牌文案缩短。
- 玩家下兵按点击位置稳定落位。
- 防止中央点击却刷到左右两侧的偏移问题。

### 8. 河道碰撞和走桥约束

最近一轮为了让兵线表现更合理，加入了：
- 单位跨河前先去桥口。
- 河道区域的碰撞约束。
- 单位位置统一经由 `constrain_unit_position()` 纠正。

这样普通地面单位不会再直接走进河里。

## 当前代码结构

### `scripts/main.gd`

职责：
- 战场常量与卡牌定义。
- UI 创建和更新。
- 玩家部署和敌方 AI 出兵。
- 兵线路径与桥梁导航。
- 单位、建筑索敌辅助。
- 音效创建与播放。
- 贴图 / SVG 资源缓存。
- 实体注册、销毁和整局胜负流程。

它现在是项目里的主控制器。随着项目继续增长，未来最值得拆分的就是它。

### `scripts/battle_unit.gd`

职责：
- 单位基础属性。
- 索敌与攻击流程。
- 兵线推进、绕行、阻挡。
- 受击、击退和硬直。
- 远程弹道飞行。
- 单位自身绘制和动画曲线。

可以把它看成轻量版战斗 Actor。

### `scripts/battle_building.gd`

职责：
- 塔的攻击和受击。
- 国王塔休眠 / 唤醒行为。
- 建筑弹道逻辑。
- 建筑自身绘制。
- 死亡目标引用清理。

## 关键实现决策

### 1. 原型阶段优先脚本绘制

当前单位和塔大量依赖 `_draw()` 做表现，而不是一开始就拆成复杂场景层级。

优点：
- 改玩法快。
- 调参数快。
- 原型阶段成本低。

缺点：
- 后续接正式动画和量产美术时，表现层需要重构一部分。

### 2. 资源缓存优先于频繁重建

项目里已经有两类缓存是被证明值得保留的：
- 纹理缓存：避免重复加载图标资源。
- 卡牌样式缓存：避免 UI 每帧创建新的 `StyleBoxFlat`。

这类优化虽然不大，但对长时间运行时的稳定性很重要。

### 3. 实体销毁后的目标清理必须做广播

曾经有一轮严重问题发生在“塔被拆那一刻”：
- 单位继续持有旧目标引用。
- 命中回调继续访问已销毁对象。
- 调试器直接中断，看起来像卡死。

当前的处理方式是：
- `battle_unit.gd` / `battle_building.gd` 每帧先清理失效引用。
- `main.gd` 的 `on_entity_destroyed()` 会通知全场实体清理指向死亡对象的引用。
- 命中结算前先用局部变量快照目标。

这是目前最重要的稳定性经验之一。

### 4. 动画曲线数值必须做保护

之前出现过单位受击时异常放大和闪烁，根因是：
- `take_damage()` 里给了 `_recovery_timer`。
- 但 `_recovery_duration` 在某些路径上仍接近 0。
- `_draw()` 中按比例换算时放大了数十倍。

当前修复方式：
- 受击时给正常的 `_recovery_duration`。
- 绘制时把 `windup_t`、`recovery_t`、`stun_t` 都夹到 `0..1`。

后续凡是继续加动画表现，都建议先考虑数值边界。

## 最近修过的重要问题

### 1. 攻击前摇卡住不出手

早期问题：
- 前摇结束后没有稳定进入真正伤害结算。
- 单位和塔会贴住目标却不攻击。

后面已经改成更稳定的推进顺序，并统一了单位和塔的解算逻辑。

### 2. 塔被拆时的空引用卡死

现象：
- 一座塔被拆后，运行像突然卡死。

根因：
- 目标引用残留。
- 销毁回调和命中回调之间存在时序竞态。

现状：
- 目标清理和销毁广播都已补上。

### 3. 弹道只显示不伤害

根因：
- 真实命中依赖 `_pending_target`，但它在发射后会被清空。

现状：
- 已改成独立的 `_projectile_target_node` 跟踪真实命中对象。

### 4. 手机手牌布局过长

根因：
- `Mini P.E.K.K.A` 的名称在窄屏按钮里过长。

现状：
- 手牌 UI 使用更短的 `ui_name`。

### 5. 中路部署位置不稳定

根因：
- 部署时横向位置带随机偏移，点击中间也可能偏到本路两侧。

现状：
- 玩家部署位置改成基于点击点稳定夹取。

### 6. 单位直接走进河里

根因：
- 原先只有兵线路径，没有地形约束。

现状：
- 已补桥梁导航和河道碰撞约束。

## 当前已知限制

### 1. `main.gd` 仍然过重

虽然已经做过实体缓存和 UI 缓存，但 `main.gd` 仍然承担了太多职责。后续如果继续扩内容，建议优先拆出：
- 卡牌 / 牌组管理。
- 战场导航与路径规则。
- 实体注册表。
- UI 控制层。

### 2. AI 仍然较基础

当前 AI 主要还是：
- 检查可支付卡牌。
- 随机选卡。
- 随机选路。

它还没有真正做出反制、节奏控制和攻守判断。

### 3. 表现层仍然是原型级

虽然已经比最初的圆形占位图好很多，但还没有：
- 正式角色动画。
- 死亡特效。
- 更完整的命中和桥头会战表现。
- 正式量产地形美术。

### 4. Android 真机验证仍然不足

项目已经补了竖屏、图标和导出配置，也修过资源加载链路，但某些移动端表现仍然更适合在真机上反复确认，尤其是：
- 图标显示。
- 触屏部署手感。
- 桥口与河道约束。
- 长时间运行时的性能和发热。

## 建议的下一步顺序

优先级 1：继续做真机验证和导出链路稳定化。
- 确认 Android 图标、触控和资源加载都稳定。
- 如果 SVG 在 Android 上仍不稳，考虑逐步换成 PNG 导出资源。

优先级 2：继续拆分 `main.gd`。
- 优先拆牌组管理和导航规则。
- 降低后续改动的耦合风险。

优先级 3：增强 AI 和内容深度。
- 加更多卡牌。
- 增强出牌策略。
- 做更明确的单位差异化。

优先级 4：继续打磨表现层。
- 更完整的动作、死亡反馈和法术特效。
- 更细的桥口拥堵和碰撞表现。
